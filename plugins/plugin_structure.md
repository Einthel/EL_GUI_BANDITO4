# Структура и требования к плагинам El_GUI_Bandito

Документация описывает архитектурные стандарты для разработки новых модулей (плагинов). За эталон взят плагин `shortcut`.

## Шаг 1: Анатомия файловой структуры

Каждый плагин должен быть полностью автономным и располагаться в собственной директории `plugins/[plugin_id]/`. Это необходимо для корректной работы системы OTA-обновлений и менеджера плагинов.

### Рекомендуемая структура папок:

```text
plugins/[plugin_id]/
├── [plugin_id]_manifest.json     # Паспорт плагина (обязательно)
├── [plugin_id]_bandito.py        # Серверная логика и редактор (обязательно)
├── [plugin_id]_cliento.py        # Клиентская логика для RPi (обязательно)
├── src/                          # Вспомогательные Python-модули
├── config/                       # JSON-конфигурации, стили, пресеты
└── resources/                    # Статические ресурсы
    ├── ui_raw/                   # Исходные .ui файлы (Qt Designer)
    ├── ui_done/                  # Скомпилированные .py файлы интерфейса
    ├── ico/                      # Иконки (png, svg, ico)
    └── sound/                    # Звуковые эффекты (wav, mp3)
```

### Ключевые требования к расположению:

*   **Автономность:** Плагин не должен зависеть от файлов других плагинов. Все специфичные для него ресурсы (иконки, звуки, стили) должны находиться внутри его папки.
*   **Разделение UI:** 
    *   `.ui` файлы хранятся в `resources/ui_raw/`.
    *   Скомпилированные версии в `resources/ui_done/`. 
    *   *Важно:* Клиент (RPi) не компилирует UI, он скачивает только готовые `.py` файлы из `ui_done`.
*   **Именование файлов:** Главные файлы логики должны иметь суффиксы `_bandito` (сервер) и `_cliento` (клиент). Это позволяет ядру автоматически идентифицировать роли файлов.
*   **Конфигурации:** Все изменяемые данные (настройки кнопок, пути к звукам, CSS-стили) должны быть вынесены в папку `config/` в формате JSON.
*   **Ресурсы:** Папки `ico/` и `sound/` внутри `resources/` позволяют системе обновлений автоматически синхронизировать медиа-контент между устройствами.

## Шаг 2: Спецификация Манифеста (`_manifest.json`)

Манифест — это «паспорт» плагина, необходимый для его идентификации менеджером плагинов и корректной работы системы OTA-обновлений. Файл должен называться `[plugin_id]_manifest.json` и располагаться в корне папки плагина.

### Шаблон манифеста:

```json
{
  "id": "my_plugin",
  "name": "My New Plugin",
  "version_file": "plugins/my_plugin/my_plugin_manifest.json",
  "min_app_version": "0.0.1.0",
  "icon": "plugins/my_plugin/resources/ico/plugin_icon.png",
  "description": "Краткое описание функционала плагина",
  "author": "YourName",
  "directories_to_ensure": [
    "plugins/my_plugin/resources/ui_done/",
    "plugins/my_plugin/config/",
    "plugins/my_plugin/resources/ico/"
  ],
  "files_map": [
    {
      "remote_path": "plugins/my_plugin/my_plugin_cliento.py",
      "local_dir": "plugins/my_plugin"
    },
    {
      "remote_path": "plugins/my_plugin/resources/ui_done/ui_my_plugin_cliento.py",
      "local_dir": "plugins/my_plugin/resources/ui_done"
    },
    {
      "remote_path": "plugins/my_plugin/resources/ico",
      "local_dir": "plugins/my_plugin/resources/ico",
      "is_directory": true
    }
  ]
}
```

### Ключевые поля и правила:

*   **`id`**: Уникальный технический идентификатор плагина (совпадает с именем папки).
*   **`min_app_version`**: Единственный источник правды для версии. Инкремент этого поля на сервере запускает обновление на клиентах.
*   **`version_file`**: Всегда должен указывать на сам файл манифеста. Это гарантирует атомарность обновления версии.
*   **`directories_to_ensure`**: Список путей, которые клиентский апдейтер создаст перед загрузкой файлов.
*   **`files_map`**: Карта синхронизации файлов:
    *   `remote_path`: Путь к файлу/папке на сервере (от корня проекта).
    *   `local_dir`: Папка назначения на клиенте.
    *   `is_directory: true`: Флаг для рекурсивного сканирования папки сервером. Позволяет синхронизировать все иконки или звуки без их перечисления в списке.
*   **Пути**: Все пути в манифесте должны быть относительными от корня рабочего пространства (начинаться с `plugins/`).

## Шаг 3: Требования к серверной части (`*_bandito.py`)

Серверный модуль — это «мозг» плагина на стороне ПК. Он должен обеспечивать три базовых процесса: **Встраивание**, **Исполнение** и **Синхронизацию**.

### Ключевые тезисы:

*   **Наследование от `QWidget`:** Плагин обязан быть виджетом, чтобы ядро могло встроить его интерфейс (редактор/панель управления) в `right_frame` сервера.
*   **Архитектурное разделение (Manager & Service):** Аналогично клиенту, серверная часть должна быть разделена на уровни:
    *   `xx_bandito.py` (UI): Только отображение и сигналы пользователя.
    *   `xx_bandito_manager.py` (Логика): Управление данными (конфиги, префабы), принятие решений и вызов синхронизации.
    *   `xx_bandito_service.py` (Данные): Низкоуровневые операции (I/O, сетевые запросы к API broadcast, выполнение системных команд).
*   **Изоляция ресурсов (UI):** Импорт скомпилированного UI должен быть защищенным (через `importlib`), чтобы пути к иконкам и ресурсам внутри одного плагина не конфликтовали с другими.
*   **Входная точка `plugin_path`:** В конструктор (`__init__`) всегда передается абсолютный путь к папке плагина. Используйте его для доступа к своим конфигам и ресурсам.
*   **Метод `handle_button_press` (Обязательно):** Универсальный обработчик событий от клиента. Ядро вызывает этот метод при любом взаимодействии пользователя с пультом (нажатие кнопки, выбор пресета).
*   **Механизм Broadcast (Синхронизация):** Если данные на сервере изменились (настройки, состояние), плагин обязан сам инициировать отправку команды клиентам через POST-запрос на `http://127.0.0.1:8000/api/broadcast`.
*   **Интеграция с Core-сервисами:** Плагин должен использовать общие менеджеры (`ElSoundManager`, `ConfigManager`) для работы со звуком и системными настройками.

## Шаг 4: Интеграция с ядром (`el_core`)

Плагины не должны работать в полной изоляции. Для стабильности и предсказуемости системы они обязаны взаимодействовать с ядром через стандартные механизмы.

### Основные точки интеграции:

1.  **Звуковая подсистема (`ElSoundManager`):**
    *   Плагин не должен создавать свои плееры. Он инициализирует `ElSoundManager(self)` и загружает в него конфиг звуков.
    *   Это гарантирует, что при выключении звука в главных настройках сервера, плагин тоже "замолчит".

2.  **Сетевое взаимодействие (`ElComManager` / API):**
    *   Плагин не управляет WebSocket-соединением напрямую. 
    *   Для рассылки обновлений используется единая точка входа — локальный HTTP API сервера. Это позволяет избежать конфликтов при одновременной работе нескольких плагинов.

3.  **Управление состоянием (`ElStateManager`):**
    *   Ядро отслеживает, какой плагин сейчас активен. Плагин может запрашивать у ядра текущую конфигурацию слотов или статус подключения клиентов.

4.  **Конфигурация (`ConfigManager`):**
    *   Для чтения общих настроек (IP, порты, флаги Dev Mode) плагины используют стандартный `ConfigManager`, указывая путь к глобальным конфигам в папке `configs/`.

> **Важно:** Не нужно писать свой API для базовых вещей (звук, сеть, конфиги). Используйте готовые классы из `el_core` и `src/`. Свой API стоит создавать только для специфических внешних интеграций (например, управление плеером Yandex Music).

## Шаг 5: Требования к клиентской части (`*_cliento.py`)

Клиентский модуль — это интерфейс плагина на Raspberry Pi. Он должен быть максимально **легким**, **отзывчивым** и **синхронным** с сервером.

### Ключевые тезисы:

*   **Инициализация с сокетом:** В конструктор (`__init__`) передается `socket_client`. Плагин обязан подписаться на сигнал `message_received` через свой менеджер или напрямую, чтобы реагировать на команды сервера (например, `SHORTCUT_CONFIG_UPDATE`).
*   **Динамический UI:** Если плагин подразумевает сетку кнопок или страниц, они должны генерироваться процедурно на основе JSON-конфига. Это позволяет менять интерфейс без обновления кода.
*   **Визуальный отклик (Анимация):** Для работы на тачскрине обязательна анимация нажатия (масштабирование или изменение прозрачности) через `QPropertyAnimation`. Параметры анимации рекомендуется выносить в `style_shortcut_cliento.json`.
*   **Оптимизация (EventFilter):** Используйте `eventFilter` для централизованной обработки нажатий и анимаций. Это снижает нагрузку на CPU Raspberry Pi по сравнению с созданием сотен индивидуальных сигналов.
*   **Локальный звук:** Плагин должен воспроизводить звуки кликов локально через `ElSoundManager`, не дожидаясь подтверждения от сервера, для минимизации задержки (Input Lag).
*   **Отправка команд:** Любое взаимодействие (нажатие кнопки) должно отправлять на сервер команду `PLUGIN_BUTTON_PRESS` с уникальным ID действия.
*   **Сброс состояний:** При переключении страниц или плагинов необходимо принудительно сбрасывать фокус и hover-эффекты (`WA_UnderMouse`), так как тачскрины часто "замораживают" эти состояния.

## Шаг 6: Архитектурное разделение (Manager & Service)

Для поддержания чистоты кода и разделения ответственности (Separation of Concerns), клиентская часть плагина должна быть разделена на три уровня: **UI** (`cliento.py`), **Логика** (`manager.py`) и **Данные** (`service.py`).

### 1. `xx_cliento_service.py` (Уровень I/O и данных)
Содержит только чистые функции без состояния (stateless) для работы с файловой системой.
*   **Загрузка/Сохранение:** Чтение и запись JSON-конфигов, стилей и анимаций.
*   **Обработка ресурсов:** Декодирование и сохранение иконок (`base64`), поиск путей к звукам.
*   **Парсинг:** Преобразование JSON-структур в CSS-строки для Qt.
*   **Цель:** Изолировать UI от работы с файлами и обработки исключений `try/except`.

### 2. `xx_cliento_manager.py` (Уровень бизнес-логики)
Координатор плагина, управляющий его состоянием и связью с сервером. Наследуется от `QObject`.
*   **Диспетчер сообщений:** Принимает данные от сокета (`on_server_message`) и распределяет их (обновление конфига, сохранение иконки).
*   **Управление состоянием:** Хранит текущую конфигурацию в RAM и уведомляет UI об изменениях через сигналы (например, `config_updated`).
*   **Принятие решений:** Определяет тип действия при нажатии (локальное системное действие или отправка команды на сервер).
*   **Цель:** Сделать логику плагина независимой от графического интерфейса и удобной для отладки.

### 3. `xx_cliento.py` (Уровень UI)
Отвечает исключительно за отображение и взаимодействие с пользователем.
*   **Отрисовка:** Создание виджетов, применение стилей и анимаций.
*   **События:** Перехват нажатий и передача их в `manager`.
*   **Цель:** Обеспечить максимально быструю и плавную отрисовку интерфейса.
